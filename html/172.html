lua scripting support (eg. custom conditons) #172

  • Enhancment
  • Fixed
Closed
Assigned to
Cybeloras
  • Twister created this issue Apr 18, 2011

    I would like to see scripting support added. The Idea came up to me uppon reading this thread where someone 1st posted a Tukui_Filger patch, and later another one took the tiny patch and created a Weak Auras config and a pitbull script from it. In the case of this thread, the users are interested in information displayed in a buffs tooltip. This is the posted lua script for pitbull

    local ES = CreateFrame( "GameTooltip", "ES", nil, "GameTooltipTemplate" );
ES:SetOwner( WorldFrame, "ANCHOR_NONE" );
local eson = UnitBuff("player", "Empowered Shadow")
if eson then
ES:SetUnitBuff("player", eson)
    local text = ESTextLeft2:GetText()
    text = string.match(text, "%d+")
    return text   
end
ES:Hide()

    This script could be easily modified to return true or false to show above or below a certain trashhold. This would be the way i would think about custom conditions: An abstract function the user could implement which should return an boolean value to determine whether the condition is true or false.

  • Cybeloras posted a comment Apr 18, 2011

    Funny that you should post this, because I have been meaning to do this for a while. The only reason that I haven't is because I'm not sure how user-friendly it needs to be. Currently, this is what a typical condition function looks like after it has been compiled:

    if not ((UnitHealth("pet")/UnitHealthMax("pet") < 0.61)or ((GetPetHappiness() or 0) < 3)and(nil == UnitIsDeadOrGhost("pet"))) then
		TellMeWhen_Group7_Icon6:SetAlpha(0)
		return true
	end

    All condition functions have their metatable set to TMW.CNDT.Env, so any variables or functions, including the icon TellMeWhen_Group7_Icon6, must be defined there. Simple conditions such as that one are created from strings, such as this one for health:

    [[UnitHealth(c.Unit)/UnitHealthMax(c.Unit) c.Operator c.Level]]

    Now, some conditions make use of helper functions because they do more complex things than can be done inside an "if (...) then ... end" check. here is one such condition (the buff condition):

    [[AuraStacks(c.Unit, c.NameName, "HELPFUL") c.Operator c.Level]]
--------------------------------------------------------------------------------------------------------
local function AuraStacks(unit, name, filter)
	local buffName, _, _, count = UnitAura(unit, name, nil, filter)
	if not buffName then
		return 0
	elseif buffName and count == 0 then
		return 1
	else
		return count
	end
end

    You example is one such case where an external helper function would be extremely helpful (but not required, here is that functionality in a normal condition format (wrap it in [[]] to make it a string, I didn't here for syntax highlighting):

    not CreateFrame( "GameTooltip", "ES", nil, "GameTooltipTemplate" ):SetOwner( WorldFrame, "ANCHOR_NONE" ) and not ES:SetUnitBuff("player", (UnitBuff("player", c.NameName) or "")) and (tonumber(strmatch((ESTextLeft2:GetText() or ""), "%d+")) c.Operator c.Level) and not ES:Hide()

    ) But obviously, that has horrible readability, and it also creates a new frame every single time the condition is checked. The ultimate question is this: should users create helper functions externally with something like _DevPad or Hack (and then store them in TMW.CNDT.Env), or should I allow users to make their custom condition in the format of a regular function ("if (...) then return true else return false end") and then have the addon create and manage these for the user, or should I implement some sort of extensive Lua editor into TMW that allows them to define and associate functions with custom conditions. Personally, I am leaning towards the first option for a few reasons:

    1. It allows users to manage their own efficiency (not creating a new frame every call, just scope one into the function, they can also upvalue whatever they need, etc.)
    2. It is easiest for me to implement
    3. It sticks with the existing format of conditions and condition functions, so I don't really have to do anything. (Isn't this the same as #2? I just wanted three points)
    4. I worded the last 2 choices pretty poorly.
